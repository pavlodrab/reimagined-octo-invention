# План обновления: Гибкая система оценки с уникальными баллами и обязательным голосованием

## Обновления к предыдущему плану (plan2.md)

### 1. Настраиваемая шкала оценки
- Администратор может задать максимальное значение шкалы (например, 10, 15, 20) через админские настройки Mini App.
- Минимальное значение всегда 0 (или 1, в зависимости от выбора; по умолчанию 0–10).
- Шкала сохраняется в конфигурации бота (`config.json` или в таблице `settings` в SQLite) как `max_rating`.
- При изменении шкалы уже сохранённые голоса pozostają без изменений (можно добавить опцию конвертации или сброса).

### 2. Уникальная оценка от каждого пользователя внутри одного матча
- При голосовании за конкретный матч каждый пользователь должен присвоить **разные** числовые значения каждому игроку из списка (например, если шкала 0–10 и игроков 11, то каждой оценке должно соответствовать уникальное число).
- Реализация:
  - При открытии вкладки голосования система показывает список игроков текущего матча.
  - Рядом с каждым игроком – поле ввода (или селектор) для выбора балла.
  - Выбранное значение блокируется для остальных игроков (нельзя повторить).
  - Если пользователь пытается ввести уже использованное число – появляется подсказка: «Этот балл уже использован, выберите другой».
  - Если шкала шире, чем количество игроков, пользователь может оставлять некоторые баллы неиспользованными.
- Это гарантирует ранжирование игроков пользователем без ничьих.

### 3. Оценка засчитывается только после нажатия кнопки «Отправить»
- Пока пользователь не нажал кнопку «Голосовать» (или «Отправить»), его выборы не сохраняются.
- Возможность отмены выбора или сброса всех полей перед отправкой.
- После нажатия:
  - Система проверяет, что за **каждого** игрока из списка выбран балл (обязательное голосование).
  - Если хотя бы один игрок не имеет оценки – показывается сообщение: «Пожалуйста, оцените всех игроков перед отправкой».
  - После подтверждения все выбранные значения отправляются одним запросом на бэкенд (`/api/vote_batch`).
  - Бэкенд сохраняет каждый голос как отдельную запись с timestamp и user_id, но также помечает, что они относятся к одному «раунду» голосования (можно добавить поле `vote_round_id` или группировать по `match_id + user_id + timestamp` внутри одного секунды).

### 4. Обязательное голосование за всех игроков перед отправкой
- Как указано выше, отправка блокируется, пока не выбраны оценки для всех игроков.
- Индикатор прогресса: например, «Оценено X из Y игроков».
- При попытке отправить незавершённый голос – подсвечиваются незаполненные поля и показывается подсказка.

### 5. Изменения в API бэкенда
#### Новый эндпоинт
- `POST /api/vote_batch`
  - Тело:
    ```json
    {
      "match_id": int,
      "user_id": int,
      "votes": {
        "player_id_1": rating_int,
        "player_id_2": rating_int,
        ...
      }
    }
    ```
  - Проверки на стороне сервера:
    - Пользователь действительно участник (можно пропустить, если доверяем initData).
    - Для каждого player_id в списке матча существует запись.
    - Все оценки находятся в диапазоне `[0, max_rating]`.
    - Оценки уникальны среди переданных (чтобы предотвратить подделку).
  - Сохранение: каждая пара (player_id, rating) сохраняется как отдельная запись в таблице `votes` с полями:
    - `id`, `match_id`, `user_id`, `player_id`, `rating`, `timestamp`, `batch_id` (генерируемый UUID для этого отправления).

#### Существующие эндпоинты (можно оставить для совместимости)
- `POST /api/vote` – оставить как есть для одиночного голосования (может быть полезно для тестов или если админ захочет включить «свободный» режим). Но основной режим – batch.

### 6. Хранение в SQLite
- Таблица `matches`: id, home_team, away_team, match_date, status, sstats_fixture_id.
- Таблица `players`: id, match_id, name, number, position, photo_url, sstats_player_id.
- Таблица `votes`: id, match_id, user_id, player_id, rating, timestamp, batch_id.
- Таблица `settings`: key, value (храним max_rating, voting_period_hours, sstats_token, owner_id, default_background, etc.).
- Таблица `admin_logs`: id, admin_user_id, target_user_id, action (add_admin, remove_admin, change_rating, change_background, etc.), details_json, timestamp.
- Таблица `user_backgrounds`: user_id, background_url (персональный фон), is_default (bool).

### 7. Админские возможности (расширенные)
- Назначение/удаление администраторов через вкладку «Админ-панель» (visible only if user_id == owner_id or listed in admins table).
- При назначении нового админа запись в `admin_logs`.
- Возможность изменить шкалу評価 (max_rating) – сохраняется в `settings`.
- Возможность глобально изменить фон приложения (загрузить изображение, которое будет использоваться для всех пользователей, если они не выбрали персональный).
- Персональный фон: пользователь может загрузить своё изображение в настройках; оно сохраняется в `user_backgrounds` и применяется только к его виду.
- Журнал изменений голосов: админ может вручную корректировать оценку конкретного пользователя за конкретного игрока (добавить/убрать балл) – запись сохраняется в `votes` (перезаписываем или добавляем корректирующую запись с меткой type='adjustment') и в `admin_logs` с деталями.
- Отображение текущего количества голосов за каждого игрока (сумма, среднее) в админ-панели.

### 8. UI/UX изменения в Mini App
- На вкладке голосования:
  - Выпадающий список или поле для выбора шкалы (читается из настроек, но если админ разрешил менять на лету – можно показывать текущее значение).
  - Селекторы оценки (можно использовать радиокнопки или выпадающий список с доступными числами; после выбора число удаляется из списка остальных).
  - Индикатор «Осталось оценить: X игроков».
  - Кнопка «Отправить» неактивна, пока не выбраны все оценки.
  - После отправки – показать подтверждение и обновить средние оценки (можно запросить результаты через `/api/results/{match_id}`).
- На вкладке настроек:
  - Поле для выбора шкалы оценки ( admins only ).
  - Превью фона и кнопки загрузки.
  - Переключатель персонального фона.
  - Язык, уведомления, тема (светлая/тёмная).
- Админ-панель (отдельная вкладка, видимая только админам):
  - Список текущих админов с кнопками «Удалить права».
  - Поле ввода username или user_id для назначения нового админа.
  - Журнал действий (таблица с последними записями из `admin_logs`).
  - Секция «Корректировка голосов»: выбор пользователя, выбора матча, выбора игрока, ввод корректировки (±1, ±2 или установить конкретное значение) и кнопка «Применить».
  - Показ текущей шкалы評ка и возможность её изменить.

### 9. Последовательность действий пользователя
1. Открывает Mini App через бота.
2. Видит вкладку «Голосовать за текущий матч».
3. Система определяет текущий матч (по расписанию или через API sstats.net) и выводит список игроков Челси.
4. Пользователь присваивает каждому игроку уникальный балл из доступного диапазона (например, 0–15).
5. По мере выбора баллы исчезают из списка доступных для остальных игроков.
6. Когда все игроки оценены, кнопка «Отправить» становится активной.
7. Пользователь нажимает «Отправить» → данные отправляются пакетом на бэкенд.
8. Бэкенд сохраняет голосы, возвращает успех.
9. Mini App показывает уведомление «Голос accepted!» и, опционально, обновляет средние оценки.
10. Пользователь может перейти на другие вкладки (прочие голосования, профиль, настройки) и увидеть обновлённую статистику.

### 10. Безопасность и проверки
- На бэкенде проверять, что запрос пришёл от авторизованного пользователя (проверять `initData` от Telegram, если запрос включает её заголовок; либо доверять, что запрос пришёл только от Mini App, который сам получает идентификатор пользователя через `WebApp.initData` и передаёт его в теле запроса).
- Защита от повторной отправки одного и того же batch: можно сохранять `batch_id` и отклонять дубликаты.
- Ограничение частоты голосования: один пользователь может отправить только один batch per match (или можно разрешить изменение голоса в течение периода, но тогда нужно обновлять запись).
- Админские действия логируются.
